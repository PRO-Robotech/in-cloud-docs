ThreeNodeKubeApi

This ETCD cluster variant is deployed on ControlPlane cluster nodes, one per each.
First, set the variables:
  • ETCD_NODE_NAME_N - ETCD pod name (Usually formatted as `etcd-CONTROLPLANE_NODE_NAME`)
  • K8S_NODE_NAME_N - K8s cluster node name where ETCD is running.
  • K8S_NODE_ADDRESS_N - K8s cluster node IP address where ETCD is running.
  • ETCD_CLIENT_PORT - ETCD cluster client connection port.
  • ETCD_PEER_PORT - Port for inter-node communication within the ETCD cluster.
  • K8S_NODE_USERNAME - Username for SSH connection to K8s cluster nodes.
    • export ETCD_NODE_NAME_1=etcd-in-cloud-k8s-sandbox-dev-csko-1-tcgzn-7plhz
    export ETCD_NODE_NAME_2=etcd-in-cloud-k8s-sandbox-dev-csko-1-tcgzn-h9g49
    export ETCD_NODE_NAME_3=etcd-in-cloud-k8s-sandbox-dev-csko-1-tcgzn-wps6k
    export K8S_NODE_NAME_1=10.10.10.3
    export K8S_NODE_NAME_2=10.10.10.4
    export K8S_NODE_NAME_3=10.10.10.5
    export K8S_NODE_ADDRESS_1=10.10.10.3
    export K8S_NODE_ADDRESS_2=10.10.10.4
    export K8S_NODE_ADDRESS_3=10.10.10.5
    export ETCD_CLIENT_PORT=2379
    export ETCD_PEER_PORT=2380
    export K8S_NODE_USERNAME=admin
    Note

    In this case, if when starting the restored ETCD cluster node, its configuration contains connection endpoints and names of other nodes that are not configured to work with the restored node, then the restored node's pod may crash with an error until the correct configuration appears on the other nodes.

    Steps 1 through 8 are recommended to be performed synchronously on all three ETCD cluster nodes.

    • Master-1
    • Master-2
    • Master-3
    1. Upload the snapshot file to the first cluster node VM at /tmp/etcd/backup.db using rsync
    ssh capv@${K8S_NODE_ADDRESS_1} "mkdir -p /tmp/etcd"
    rsync --rsync-path="sudo rsync"         LOCAL_PATH_TO_BACKUP/backup.db         capv@${K8S_NODE_ADDRESS_1}:/tmp/etcd/backup.db
    1. Connect to the first cluster node VM via ssh and switch to superuser. Then stop kubelet to prevent it from starting ETCD.
    systemctl stop kubelet
    1. Stop and remove the etcd container:
    export CONTAINER_ID="$(crictl ps -a          --label io.kubernetes.container.name=etcd         --label io.kubernetes.pod.namespace=kube-system         | awk 'NR>1{r=$1} $0~/Running/{exit} END{print r}')"
    crictl stop ${CONTAINER_ID}
    crictl rm ${CONTAINER_ID}

    1. Create a backup.

    2. Remove old files from the directory /var/lib/etcd

    rm -rf /var/lib/etcd/*
    1. Prepare the DB from the snapshot:
    etcdctl snapshot restore /tmp/etcd/backup.db         --name=${K8S_NODE_1}         --data-dir /var/lib/etcd         --initial-advertise-peer-urls=https://${K8S_NODE_ADDRESS_1}:2380         --initial-cluster=${K8S_NODE_1}=https://${K8S_NODE_ADDRESS_1}:2380,${K8S_NODE_2}=https://${K8S_NODE_ADDRESS_2}:2380,${K8S_NODE_3}=https://${K8S_NODE_ADDRESS_3}:2380
    note

    Note that the local etcdctl file is used and the snapshot is also located on the master in the directory /tmp/etcd. The data-dir specifies the host directory mounted inside the ETCD container that will be used for storing DB files.

    1. Check and if necessary edit the ETCD startup parameters in the file $/etc/kubernetes/manifests/etcd.yaml specified in the spec.containers[0].command section. Specifically, check for the presence of required fields and their values; fields not in this list should not be deleted!
    --advertise-client-urls=https://10.10.10.3:2379
    --cert-file=/etc/kubernetes/pki/etcd/server.crt
    --client-cert-auth=true
    --data-dir=/var/lib/etcd
    --key-file=/etc/kubernetes/pki/etcd/server.key
    --listen-client-urls=https://127.0.0.1:2379,https://10.10.10.3:2379
    --listen-metrics-urls=http://0.0.0.0:2381
    --listen-peer-urls=https://10.10.10.3:2380
    --name=in-cloud-k8s-sandbox-dev-csko-1-tcgzn-7plhz
    --peer-cert-file=/etc/kubernetes/pki/etcd/peer.crt
    --peer-client-cert-auth=true
    --peer-key-file=/etc/kubernetes/pki/etcd/peer.key
    --peer-trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
    --trusted-ca-file=/etc/kubernetes/pki/etcd/ca.key
    ParameterDescription
    advertise-client-urls Advertised connection endpoints for client connections. Matches the K8s cluster node IP address
    cert-file Path to the ETCD cluster certificate
    client-cert-auth Enable/disable certificate-based user authorization
    data-dir Directory where ETCD DB files are stored
    key-file Path to the ETCD cluster certificate key
    listen-client-urls Endpoint where ETCD will accept client requests (0.0.0.0, 127.0.0.1, or K8s cluster node IP address)
    listen-metrics-urls Endpoint where ETCD will accept metrics requests (0.0.0.0, 127.0.0.1, or K8s cluster node IP address)
    listen-peer-urls Endpoint where ETCD will accept requests from other ETCD cluster nodes (0.0.0.0, 127.0.0.1, or K8s cluster node IP address)
    name ETCD cluster node name
    peer-cert-file Path to the certificate for inter-node ETCD cluster connections
    peer-client-cert-auth Path to the certificate for authorizing inter-node ETCD cluster connections
    peer-key-file Path to the certificate key for inter-node ETCD cluster connections
    peer-trusted-ca-file Path to the trusted root certificate for inter-node ETCD cluster connections
    trusted-ca-file Path to the trusted root certificate of the ETCD cluster
    note

    If the initial-cluster-state parameter equals "existing" or there are no DB files and metadata, then all other parameters whose names start with initial will be ignored. In our case, we have already restored the DB files and added metadata during the DB preparation from the snapshot.
