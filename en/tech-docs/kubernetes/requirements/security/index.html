2.10 Information Security

2.10.1 Identification and Authentication Requirements​

No.Requirement
1A unique account is created for each user
2Authentication mechanisms are implemented using secure authentication protocols. Password confidentiality during storage and transmission is ensured by encryption or hashing with strong cryptographic algorithms
3Active Directory authentication capability is provided
4Two-factor user authentication mechanisms are implemented
5Credential verification for all accounts is performed on the server-side IS components (no offline authentication capability)
6During IS authentication, the password is not displayed during input, verification of entered information (login, password) is performed only after complete input, in case of error, the system does not specify which data was entered incorrectly

2.10.2 Information Security Event Collection and Analysis​

No.Requirement
1Security event logging capability is provided, including: identification attempts, privilege changes, user profile modifications, privileged user actions, access to protected objects
2Information is recorded for each security event: event type, date and time, identification information, event result, access subject and object, action description
3Access to security event logs is restricted to authorized users only

2.10.3 Information Protection When Using Mobile Devices​

No.Requirement
1User access is restricted using approved mobile software versions
2SSL/TLS Pinning technology is applied to protect root certificates
3Secure storage is used for storing user authentication data
4Event logging to unprotected areas is not permitted
5Transfer of confidential information between applications is restricted

2.10.4 Password Protection and Account Requirements​

No.Requirement
1Users are provided with the ability to independently set and change their password
2The IS administrator can assign an initial password with mandatory password change by the user at first authentication
3A user account unused for 30 (thirty) days is locked in the IS
4Users are notified in advance (at least 14 days) about the need to change their password (via messages, hints, or email notifications)
5Session idle timeout (no more than 15 minutes) can be configured, after which the session is forcibly locked until re-authentication
6Password history (password hashes) storage is provided to prevent reuse. The last 7 passwords are checked for repetition
7Password length and complexity restrictions can be configured (uppercase and lowercase letters, numbers, and special characters)
8Account lockout for a predetermined period after a specified number of failed authentication attempts is provided
9Mandatory password change at set intervals is enforced
10Password input complication techniques are implemented: CAPTCHA, increased wait time

2.10.5 Access Rights Management/Privilege Separation Requirements​

No.Requirement
1User authorization mechanisms with access rights separation for information/data and functions within the IS are implemented
2User access rights are provided based on attribute-based or role-based models
3Explicit restriction or prohibition of user actions in the IS is enforced until identification and authentication procedures are completed
4Direct access by users, administrators, and third parties to production IS databases is prohibited by default
5Storage and processing of Bank information in external cloud services is prohibited
6IS users are informed before directly gaining access to the IS/IA in the form of a system notification/banner. The notification or banner remains on the screen until users confirm the terms of use and take explicit action to log in to the IS

2.10.6 General Information Security Requirements​

No.Requirement
1Viewing summary information on cluster security policies, primarily Network Policy
2Supports access control based on role-based (RBAC) or attribute-based (ABAC) models
3Supports access control at the record/entity level of the same type (row-level security)
4Stored data encryption mechanism is available (specify which data and encryption algorithm requirements)
5Audit of events for configuration changes, information system settings, and update installations
6Audit of data change events resulting from business process execution and chronological user action sequence
7Ability to exclude sensitive data from event log records, such as card PAN numbers, etc.
