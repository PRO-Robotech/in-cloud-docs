> Ниже два рабочих сценария подключения worker-ноды к кластеру.
> Первый дает полный контроль над bootstrap-процессом, второй повторяет привычный путь через `kubeadm join`.

<Tabs groupId="auth-strategy">

  <TabItem value="bootstrap-token" label="Bootstrap Token (Hard Way)">

    > Это ручной сценарий: создаем `bootstrap-kubelet.conf`, запускаем kubelet и при необходимости сами проводим CSR
    > через Kubernetes API.
    > Подходит, если хотите видеть весь путь TLS Bootstrap без автоматики `kubeadm`.

    :::danger Внимание
    В примере ниже используется **статичный bootstrap-токен** для всех worker-узлов.
    Для production-среды лучше выпускать **отдельный токен на каждую ноду** и задавать ему ограниченный срок жизни.
    :::

    <h3>Создание bootstrap-токена</h3>

    > Bootstrap-токен хранится в Secret в namespace `kube-system` и дает новой ноде право начать bootstrap-процесс.
    > Ниже два способа его создать.

    <Tabs groupId="token-creation">

      <TabItem value="manual" label="Вручную">

        :::warning 🖥️ Мастер-нода
        Команды ниже выполняются на **мастер-ноде** или на хосте с kubeconfig, имеющим права на создание Secret в namespace `kube-system`.
        :::

        <h4>Переменные окружения</h4>
        <CodeBlock language="bash">
          {dedent`
            export AUTH_EXTRA_GROUPS="system:bootstrappers:kubeadm:default-node-token"
            export DESCRIPTION="kubeadm bootstrap token"
            export EXPIRATION=$(date -d '24 hours' "+%Y-%m-%dT%H:%M:%SZ")
            export TOKEN_ID="fjt9ex"
            export TOKEN_SECRET="lwzqgdlvoxtqk4yw"
            export USAGE_BOOTSTRAP_AUTHENTIFICATION="true"
            export USAGE_BOOTSTRAP_SIGNING="true"
          `}
        </CodeBlock>

        <h4>Создание Secret</h4>
        <CodeBlock>
          {dedent`
            kubectl \\
              --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/super-admin.conf \\
              apply -f - <<EOF
            ---
            apiVersion: v1
            kind: Secret
            metadata:
              name: bootstrap-token-$\{TOKEN_ID}
              namespace: kube-system
            data:
              auth-extra-groups: $(echo -n "$AUTH_EXTRA_GROUPS" | base64)
              description: $(echo -n "$DESCRIPTION" | base64)
              expiration: $(echo -n "$EXPIRATION" | base64)
              token-id: $(echo -n "$TOKEN_ID" | base64)
              token-secret: $(echo -n "$TOKEN_SECRET" | base64)
              usage-bootstrap-authentication: $(echo -n "$USAGE_BOOTSTRAP_AUTHENTIFICATION" | base64)
              usage-bootstrap-signing: $(echo -n "$USAGE_BOOTSTRAP_SIGNING" | base64)
            type: bootstrap.kubernetes.io/token
            EOF
          `}
        </CodeBlock>

      </TabItem>

      <TabItem value="kubeadm-token" label="kubeadm token create">

        :::warning 🖥️ Мастер-нода
        Команда выполняется на **мастер-ноде** или на хосте с kubeconfig, имеющим права на управление bootstrap-токенами.
        :::

        <CodeBlock language="bash">
          {dedent`
            kubeadm token create \\
              --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/super-admin.conf \\
              --print-join-command \\
              --ttl 24h
          `}
        </CodeBlock>

        :::note Вывод команды
        <CodeBlock language="bash">
          {dedent`
            kubeadm join api.${CUSTOM_VALUE.clusterName.value}.${CUSTOM_VALUE.kubernetesBaseDomain.value}:6443 \\
              --token <generated-token> \\
              --discovery-token-ca-cert-hash sha256:<hash>
          `}
        </CodeBlock>
        :::

        > Полученный токен можно использовать при создании `bootstrap-kubelet.conf` или передать в конфигурацию `kubeadm join`.

      </TabItem>

    <h3>Создание bootstrap-kubelet.conf</h3>

    :::info 🖥️ Worker-нода
    Все команды данного этапа выполняются на **worker-ноде**.
    На worker-ноде файл `ca.crt` еще отсутствует. CA-данные получаем из публичного ConfigMap `cluster-info`
    в namespace `kube-public`, доступного анонимно через kube-apiserver.
    :::

    <h4>Переменные окружения</h4>

    <CodeBlock language="bash">
      {dedent`
        export BOOTSTRAP_TOKEN=${CUSTOM_VALUE.bootstrapToken.value}
        export API_SERVER="https://api.${CUSTOM_VALUE.clusterName.value}.${CUSTOM_VALUE.kubernetesBaseDomain.value}:6443"
      `}
    </CodeBlock>

    <h4>Рабочая директория</h4>

    <CodeBlock language="bash">
      {dedent`
        mkdir -p ${CUSTOM_VALUE.kubernetesBaseFolderPath.value}
      `}
    </CodeBlock>

    <h4>Получение CA из cluster-info</h4>

    <CodeBlock language="bash">
      {dedent`
        export CA_DATA=$(curl -sk "$\{API_SERVER}/api/v1/namespaces/kube-public/configmaps/cluster-info" | \\
          jq -r '.data.kubeconfig' | \\
          grep 'certificate-authority-data' | \\
          awk '{print $2}')
      `}
    </CodeBlock>

    <h4>Сохранение CA-сертификата</h4>

    <CodeBlock language="bash">
      {dedent`
        mkdir -p ${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/pki
        echo "$\{CA_DATA}" | base64 -d > ${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/pki/ca.crt
      `}
    </CodeBlock>

    <h4>Генерация kubeconfig</h4>

    <CodeBlock language="bash">
      {dedent`
        cat <<EOF > ${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/bootstrap-kubelet.conf
        apiVersion: v1
        clusters:
        - cluster:
            certificate-authority-data: $\{CA_DATA}
            server: $\{API_SERVER}
          name: ${CUSTOM_VALUE.clusterName.value}
        contexts:
        - context:
            cluster: ${CUSTOM_VALUE.clusterName.value}
            user: tls-bootstrap-token-user
          name: tls-bootstrap-token-user@kubernetes
        current-context: tls-bootstrap-token-user@kubernetes
        kind: Config
        preferences: {}
        users:
        - name: tls-bootstrap-token-user
          user:
            token: $\{BOOTSTRAP_TOKEN}
        EOF
      `}
    </CodeBlock>

    <h3>Kubernetes CSR (имитация TLS Bootstrap)</h3>

    > Ниже показан тот же поток, который kubelet обычно проходит сам во время TLS Bootstrap:
    > ключи генерируются на worker-ноде, CSR отправляются через Kubernetes API,
    > а подпись подтверждает администратор на мастер-ноде.
    > Приватный ключ CA на worker-ноде для этого **не нужен**.

    <details>
      <summary>
        <div style={{ display: 'flex', justifyContent: 'space-between', flexWrap: 'wrap' }}>
          <p style={{ marginBottom: 0 }}>Kubelet Client Certificate (CSR)</p>
          <p className="obligatory-note-orange">● Обязателен к применению</p>
        </div>
      </summary>

      > **Назначение:** Клиентский сертификат kubelet для подключения к kube-apiserver.

      <h4>1. Генерация ключа и CSR</h4>

      :::info 🖥️ Worker-нода
      Все команды данного этапа выполняются на **worker-ноде**.
      :::

      <CodeBlock language="bash">
        {dedent`
          export HOST_NAME=worker-1
          export CLUSTER_NAME="${CUSTOM_VALUE.clusterName.value}"
          export BASE_DOMAIN="${CUSTOM_VALUE.kubernetesBaseDomain.value}"
          export FULL_HOST_NAME="${CUSTOM_VALUE.virtualMachineFullName.value}"
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          mkdir -p ${CUSTOM_VALUE.kuberneteKubeletFolderPath.value}/pki
          mkdir -p ${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/openssl/csr
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          cat <<EOF > ${CERTIFICATES.kubeletClient.crtConf}
          [ req ]
          default_bits        = ${CERTIFICATES.kubeletClient.keySize}
          prompt              = no
          default_md          = sha256
          distinguished_name  = dn

          [ dn ]
          CN = system:node:$\{FULL_HOST_NAME}
          O  = system:nodes

          [ v3_ext ]
          authorityKeyIdentifier=keyid,issuer:always
          basicConstraints=CA:FALSE
          keyUsage=keyEncipherment,dataEncipherment
          extendedKeyUsage=clientAuth
          EOF
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          openssl genrsa \\
            -out ${CERTIFICATES.kubeletClient.keyPath} ${CERTIFICATES.kubeletClient.keySize}
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          openssl req -new \\
            -key ${CERTIFICATES.kubeletClient.keyPath} \\
            -out ${CERTIFICATES.kubeletClient.csrPath} \\
            -config ${CERTIFICATES.kubeletClient.crtConf}
        `}
      </CodeBlock>

      <h4>2. Отправка CSR в Kubernetes API</h4>

      :::info 🖥️ Worker-нода
      Worker-нода аутентифицируется bootstrap-токеном через `bootstrap-kubelet.conf`.
      :::

      <CodeBlock language="bash">
        {dedent`
          export HOST_NAME=worker-1
          export CSR_NAME="$\{HOST_NAME}-kubelet-client"
          export CSR_CONTENT=$(cat ${CERTIFICATES.kubeletClient.csrPath} | base64 | tr -d '\\n')
        `}
      </CodeBlock>

      <CodeBlock>
        {dedent`
          kubectl \\
            --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/bootstrap-kubelet.conf \\
            apply -f - <<EOF
          apiVersion: certificates.k8s.io/v1
          kind: CertificateSigningRequest
          metadata:
            name: $\{CSR_NAME}
          spec:
            request: $\{CSR_CONTENT}
            signerName: kubernetes.io/kube-apiserver-client-kubelet
            usages:
            - digital signature
            - key encipherment
            - client auth
          EOF
        `}
      </CodeBlock>

      <h4>3. Утверждение CSR</h4>

      :::warning 🖥️ Мастер-нода
      Утверждение CSR выполняется на **мастер-ноде**. Укажите имя worker-ноды, для которой утверждается CSR.
      :::

      <CodeBlock language="bash">
        {dedent`
          export HOST_NAME=worker-1
          export CSR_NAME="$\{HOST_NAME}-kubelet-client"
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          kubectl \\
            --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/super-admin.conf \\
            certificate approve $\{CSR_NAME}
        `}
      </CodeBlock>

      <h4>4. Получение подписанного сертификата</h4>

      :::info 🖥️ Worker-нода
      Сертификат получаем на **worker-ноде** с использованием `bootstrap-kubelet.conf`.
      :::

      <CodeBlock language="bash">
        {dedent`
          export HOST_NAME=worker-1
          export CSR_NAME="$\{HOST_NAME}-kubelet-client"
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          kubectl \\
            --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/bootstrap-kubelet.conf \\
            get csr $\{CSR_NAME} \\
            -o jsonpath='{.status.certificate}' | base64 -d > ${CERTIFICATES.kubeletClient.crtPath}
        `}
      </CodeBlock>

      <CodeBlock language="bash">
        {dedent`
          cat ${CERTIFICATES.kubeletClient.crtPath} ${CERTIFICATES.kubeletClient.keyPath} > ${CERTIFICATES.kubeletNowClient.crtPath}
          ln -sf ${CERTIFICATES.kubeletNowClient.crtPath} ${CERTIFICATES.kubeletCurrentClient.crtPath}
        `}
      </CodeBlock>

    </details>

  <TabItem value="kubeadm" label="Kubeadm">

    > Здесь почти все делает `kubeadm join`: получает `cluster-info`, стартует kubelet и проводит TLS Bootstrap.
    > Это самый близкий к стандартной установке путь.

    <h3>Получение токена</h3>

    > Если токен не был создан заранее или истек, сгенерируйте новый.

    :::warning 🖥️ Мастер-нода
    Команда выполняется на **мастер-ноде** или на хосте с kubeconfig, имеющим права на управление bootstrap-токенами.
    :::

    <CodeBlock language="bash">
      {dedent`
        kubeadm token create \\
          --kubeconfig=${CUSTOM_VALUE.kubernetesBaseFolderPath.value}/super-admin.conf \\
          --print-join-command \\
          --ttl 24h
      `}
    </CodeBlock>

    :::note Вывод команды
    <CodeBlock language="bash">
      {dedent`
        kubeadm join api.${CUSTOM_VALUE.clusterName.value}.${CUSTOM_VALUE.kubernetesBaseDomain.value}:6443 \\
          --token <generated-token> \\
          --discovery-token-ca-cert-hash sha256:<hash>
      `}
    </CodeBlock>
    :::

    <h3>Конфигурация kubeadm</h3>

    > Полученный токен можно подставить в конфиг `kubeadm` или передать прямо в командной строке.

    :::warning Обратите внимание
    Перед запуском убедитесь, что у вас уже готов файл `kubeadm.yaml` из раздела
    [Настройка ПО (DP)](/tech-docs/kubernetes/components/all-setup-dp).
    :::

    <h4>Использование статичного конфига</h4>
    <CodeBlock language="bash">
      {dedent`
        kubeadm join \\
          --config=${CUSTOM_VALUE.kubeadmBaseConfigPath.value}/kubeadm.yaml
      `}
    </CodeBlock>

    <h4>Использование токена из командной строки</h4>
    <CodeBlock language="bash">
      {dedent`
        kubeadm join \\
          api.${CUSTOM_VALUE.clusterName.value}.${CUSTOM_VALUE.kubernetesBaseDomain.value}:6443 \\
          --token <generated-token> \\
          --discovery-token-ca-cert-hash sha256:<hash>
      `}
    </CodeBlock>
