# 2.6. Ресурсы

## 2.6.1. Требования к системе управления вычислительными ресурсами

  <thead>
    <tr>
      <th>№</th>
      <th>Требование</th>
    </tr>
  </thead>
  <tbody>

      <td>int +</td>
      <td>out +</td>
    </tr> */}
    {/* <tr>

    </tr> */}
    <tr>
      <td>3</td>
      <td>Платформа должна поддерживать возможность создания виртуальных машин на основе предустановленных шаблонов или ISO-образов операционных систем.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    {/* <tr>
      <td>4</td>
      <td>Платформа должна обеспечивать возможность создания/управления шаблонами ВМ (хранение — зона ответственности платформы виртуализации).</td>
    </tr> */}
    <tr>
      <td>5</td>
      <td>Платформа должна обеспечивать возможность создания виртуальных машин с графическим адаптером.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>6</td>
      <td>Платформа должна обеспечивать возможность подключения bare-metal графических адаптеров.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>7</td>
      <td>Платформа должна обеспечивать возможность конфигурирования платформы виртуализации для проброса графического адаптера целиком в ВМ.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>8</td>
      <td>Платформа должна обеспечивать возможность конфигурирования платформы виртуализации для проброса части графического адаптера в ВМ.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>9</td>
      <td>Платформа должна обеспечивать поддержку управления ВМ с гостевыми операционными системами Linux.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>10</td>
      <td>Платформа должна обеспечивать возможность создания отказоустойчивых кластеров для платформы контейнеризации, состоящих не менее чем из 3 узлов.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>11</td>
      <td>Платформа должна обеспечивать возможность создания отказоустойчивых кластеров для СУБД на Linux, состоящих не менее чем из 3 узлов с возможностью выбора необходимой конфигурации по каталогу.</td>
      <td>int +</td>
      <td>out +</td>
    </tr>
    <tr>
      <td>12</td>
      <td>Узел выполнения нагрузки может быть добавлен или удален из экземпляра платформы (кластера) стандартными методами, узел выполнения нагрузки может выполнять и виртуализированную нагрузку, в частности ВМ.</td>
    </tr>
    <tr>
      <td>13</td>
      <td>Платформа должна поддерживать добавление vCPU виртуальных машин без прерывания их работы.</td>
    </tr>
    <tr>
      <td>14</td>
      <td>Платформа должна поддерживать расширение RAM виртуальных машин без прерывания их работы.</td>
    </tr>
    <tr>
      <td>15</td>
      <td>Платформа должна поддерживать добавление, расширение дисков виртуальных машин без прерывания их работы.</td>
    </tr>
    <tr>
      <td>16</td>
      <td>Платформа должна поддерживать изменение параметров сетевых адаптеров виртуальных машин без прерывания их работы.</td>
    </tr>
    <tr>
      <td>17</td>
      <td>Платформа должна поддерживать добавление виртуальных сетевых адаптеров виртуальных машин без прерывания их работы.</td>
    </tr>
    <tr>
      <td>18</td>
      <td>Платформа должна поддерживать функции высокой доступности для PaaS/SaaS (High Availability).</td>
    </tr>
    <tr>
      <td>19</td>
      <td>Платформа должна поддерживать управление моментальными снимками виртуальных машин.</td>
    </tr>
    <tr>
      <td>20</td>
      <td>Платформа должна поддерживать управление моментальными снимками виртуальных дисков.</td>
    </tr>
    <tr>
      <td>21</td>
      <td>Платформа должна поддерживать функционал автоматической балансировки/оптимизации нагрузки между вычислительными узлами (Worker Nodes).</td>
    </tr>
    <tr>
      <td>22</td>
      <td>Платформа должна поддерживать обновление платформы без остановки и перебоев в работе платформы.</td>
    </tr>
    <tr>
      <td>23</td>
      <td>Платформа должна иметь API для управления и интеграции с другими системами.</td>
    </tr>
    <tr>
      <td>24</td>
      <td>Облачная платформа в части восстановления после катастроф (Disaster Recovery, DR) должна обеспечивать конфигурирование процесса восстановления после катастроф для инстансов и данных для сервисов на платформе, в том числе для реляционных и нереляционных баз данных.</td>
    </tr>
    <tr>
      <td>25</td>
      <td>Облачная платформа в части восстановления после катастроф (Disaster Recovery, DR) должна обеспечивать конфигурирование процесса восстановления после катастроф для IaaS (виртуальные машины, Microsoft Windows, Linux, блочное хранилище, программно-определяемое хранилище и т.д.).</td>
    </tr>
    <tr>
      <td>26</td>
      <td>Облачная платформа в части восстановления после катастроф (Disaster Recovery, DR) должна обеспечивать восстановление после катастроф в резервном ЦОД для подходов Warm Standby и Multi-site active/active.</td>
    </tr>
  </tbody>
</table>
